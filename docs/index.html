Podcast Agent Documentation

Architecture Overview

podcast-agent is a staged Python library and CLI for converting a source book into structured episode plans, grounded scripts, render manifests, and optional synthesized audio. This document describes the execution model, stage boundaries, canonical artifacts, and repository layout.

System Summary Execution Flow Artifacts Contracts Project Layout Runtime Notes

System Summary

Primary entrypoints Typer CLI commands and orchestration code under src/podcast_agent/pipeline.
Execution model Deterministic stages handle ingestion, indexing, manifest generation, and audio synthesis. LLM-backed agents handle structure, analysis, planning, writing, validation, and repair.
Validation boundary Stage outputs are Pydantic-validated and JSON-serializable before downstream use.
Persistence Books, chapters, chunks, embeddings, plans, scripts, reports, and repairs may be stored through the repository layer.
Output root .podcast_agent/runs/<book-title>-<timestamp>/

Execution Flow

Deterministic stage LLM-backed agent Persisted artifact Run-level log
  1. Step 1

    ingest-book

    Reads the source file, normalizes metadata, and creates BookIngestionResult.

    Artifact: ingestion.json

  2. Step 2

    structuring_agent

    Produces BookStructure with chapters, chunks, stable identifiers, and source offsets.

    Artifact: structure.json

  3. Step 3

    index-book

    Stores chunks, computes embeddings, and prepares retrieval state for later stages.

    Artifact: embeddings.json and repository rows

  4. Step 4

    analysis_agent

    Identifies themes, continuity arcs, notable claims, and candidate episode clusters.

    Artifact: analysis.json

  5. Step 5

    episode_planning_agent

    Converts analysis into SeriesPlan with episodes, beats, and claim requirements.

    Artifact: series_plan.json

  6. Step 6

    writing_agent

    Generates episode narration from the plan and retrieved source chunks.

    Artifact: script.json

  7. Step 7

    grounding_validation_agent

    Validates claim-bearing segments against cited evidence before rendering.

    Artifact: grounding_report.json

  8. Step 8

    repair_agent

    Repairs weak or unsupported sections and submits the result for revalidation.

    Artifact: repair_attempt_<n>.json when required

  9. Step 9

    render-manifest

    Creates TTS-ready render data with grounded text and SSML-ready content.

    Artifact: render_manifest.json

  10. Step 10

    synthesize-audio

    Synthesizes final episode audio from validated render output when audio is enabled.

    Artifact: final .mp3 and audio metadata in episode_output.json

run.log

Each run writes a log file alongside generated artifacts.

It records stage transitions, prompts, responses, TTS requests, and command metadata.

Contract Rules

  • Artifacts are JSON-serializable.
  • Stage outputs are schema-validated before handoff.
  • Extra fields are rejected rather than ignored.
  • Render-manifest generation is blocked until grounding passes.

Project Layout

Repository Structure

  • src/podcast_agent/agents contains stage-specific agent logic.
  • src/podcast_agent/pipeline contains orchestration code.
  • src/podcast_agent/db contains repository and persistence logic.
  • src/podcast_agent/retrieval contains embeddings and retrieval helpers.
  • src/podcast_agent/tts contains speech synthesis clients.

Runtime Notes

  • The default LLM and TTS path targets an OpenAI-compatible endpoint.
  • A heuristic adapter is available for offline tests and deterministic fallback runs.
  • Episodes target at least 30 minutes when source material supports it.
  • Short episodes are only merged away if they cannot remain above a 10-minute minimum.
  • Artifacts are written under .podcast_agent/runs/.
